### InnoDB
InnoDB 係 MySQL 預設引擎，適合大多數現代應用。
佢支援完整 ACID 交易、行級鎖定（包括 Next-Key 鎖避免幻讀）同強大 crash 恢復，90% OLTP 同 Web 系統都用佢。
優點：交易安全、高併發、支援外鍵
缺點：寫入較慢、佔用資源多，讀重場景唔及 MyISAM 快

### MyISAM
MyISAM 係舊式引擎，適合讀多寫少嘅場景。
用途主要係老系統或靜態資料表，如報表或歸檔，因鎖粒度係表級，crash 恢復弱。
優點：讀取超快、全文字搜尋支援好、空間效率高
缺點：無交易、無外鍵、crash 易損壞資料，唔適合寫入頻繁

### Memory
Memory 引擎（舊稱 HEAP）將資料全存 RAM，速度最快但非持久。
用途係臨時表、快取或 session 資料，重啟即清空，鎖粒度表級、無交易。
優點：讀寫極快、適合小量熱資料；
缺點：RAM 限制大小、無 crash 恢復、重啟丟失一切，唔適合大資料。

### Archive
Archive 引擎專為寫入為主、很少讀取嘅場景設計。
主要用長期冷數據、log 或歷史歸檔，只支援 INSERT 同 SELECT，幾乎只 append，無刪除/更新。
優點：極高壓縮率（比 InnoDB 小 80%）、寫入快；
缺點：鎖表級、無索引、查詢慢，唔支援交易或複雜操作。

## InnoDB Core Structure
- MySQL InnoDB 的核心結構分為記憶體結構與磁碟結構，.ibd 檔案與 Buffer Pool 是關鍵組成
### In Momory 記憶體結構
- InnoDB 在記憶體中快取資料，提升效能，主要包括：
  -  Buffer Pool：主要快取區，存放資料頁、索引頁、Undo 頁等，預設大小依 innodb_buffer_pool_size 設定。通常分配伺服器記憶體的 60-80%。
  -  Change Buffer：快取二級索引變更，減少隨機 I/O。
  -  Log Buffer：暫存 Redo Log，定期寫入磁碟。
  -  Adaptive Hash Index：自動建立常用索引的雜湊索引。

### On-Disk 磁碟結構
- 資料永久儲存於磁碟，主要包括：
  - .ibd 檔案：每個 InnoDB 表的獨立表空間檔案（file-per-table），存放該表資料與主索引。例：table_name.ibd。
  - 系統表空間（ibdata1）：存放 Doublewrite Buffer、共享 Change Buffer 等。
  - Redo Log：ib_logfile0/1，WAL 日誌確保耐久性。
  - Undo Log：獨立 Undo 表空間，支援 MVCC 與回滾。
  - Doublewrite Buffer：128 連續頁，防止頁寫入損壞。

## New Record Insert Flow 
1. Insert SQL
- Client executes INSERT statement through SQL Layer.
2. Data to Buffer Pool
- New record inserted into data page in Buffer Pool (memory), marked as dirty page.
3. Undo Log Generated
- Undo Log entry created for MVCC (snapshot read) and rollback capability.
4. Write to Redo Log Buffer
- Redo Log record generated in Log Buffer (physical changes for crash recovery).
5. COMMIT Confirmation
- Redo Log flushed to disk (ib_logfile), transaction commits successfully.
6. Background Flush
- Dirty pages from Buffer Pool → Doublewrite Buffer → .ibd file (delayed batch write).
```
INSERT → Buffer Pool + Undo Log + Redo Log Buffer → COMMIT (fsync Redo Log) → Background: Buffer Pool → .ibd
```

## MVCC (Multi-Version Concurrency Control)
- MVCC allows concurrent reads and writes without blocking each other.
  - Core Mechanism: Each row has multiple versions stored in Undo Log. Readers see "snapshot" data from transaction start time.
  - ReadView: Generated at SELECT start, determines visible row versions based on transaction ID.
  - Hidden Columns: Each row stores ROW_START (creation trx_id), ROW_END (deletion trx_id).
  - Implementation:
  ```text
  SELECT → ReadView (trx_id list) → Check row trx_id → Visible? → Undo Log (older version)
  ```
  - Isolation Levels: Works with REPEATABLE READ (default) and READ COMMITTED.

## InnoDB Record Lock
- Record Lock
  - Locks a specific index record.
- Gap Lock
  - Locks a range of values between existing index records (an open interval).
- Next-Level Lock
  - A Record Lock on the index record plus a Gap Lock on the gap preceding the index record.
