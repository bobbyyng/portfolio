# Java Static Typing & Compilation Flow  
# Java 強類型與編譯流程

TypeScript checks types at compile time but types are erased when it becomes JavaScript at runtime. Java keeps type discipline both at **compile time** and **runtime**, and uses a two-step process: source → bytecode → execution on the JVM.

TypeScript 編譯完變成 JS，運行時沒有類型檢查。Java 是強類型語言，編譯成 Bytecode 在 JVM 上執行，類型在編譯與運行階段都會被檢查。

## The Two-Step Compilation Flow 兩步編譯流程

Java does **not** compile directly to machine code. It uses an intermediate representation so the same code can run on any platform that has a JVM (“Write Once, Run Anywhere”).

Java 不是直接編譯成機械碼，而是先變成中間表示，讓同一份 Code 可以在任何有 JVM 的平台上運行。

So the full flow is:

**Source (.java) → javac → Bytecode (.class) → JVM (Interpreter + JIT) → Machine code on OS/hardware.**

**源碼 (.java) → javac 編譯 → 字節碼 (.class) → JVM（解釋器 + JIT）→ 在當前系統上運行的機器碼。**

## Why Bytecode? 為何要 Bytecode？

- **Portability 可移植性:** One bytecode runs on any OS (Windows, Linux, macOS) as long as a JVM exists for that platform. 同一份 Bytecode 在不同平台只要有對應的 JVM 就能運行。
- **Security & checks 安全與檢查:** JVM can verify bytecode and enforce type safety at load/run time. JVM 可以驗證字節碼並在載入/運行時執行類型檢查。
- **Optimisation 優化:** JIT can profile and optimise hot code at runtime. JIT 可以根據運行時情況對熱點做優化。

---

## Summary 小結

- **Compile time:** `javac` enforces types; wrong types prevent `.class` from being generated. 編譯階段用 javac 做類型檢查，類型不對就不會產生 .class。
- **Runtime:** JVM runs bytecode and enforces type safety (e.g. `ClassCastException` on invalid casts). 運行階段 JVM 執行字節碼並做類型檢查。
- **Flow:** `.java` → **javac** → `.class` (bytecode) → **JVM** (interpreter + JIT) → machine code. 流程：源碼 → javac → 字節碼 → JVM → 機器碼。
- **Bytecode** is the key to “Write Once, Run Anywhere” and allows the JVM to optimise with JIT. 字節碼是「一次編譯，到處運行」的基礎，也讓 JVM 可以用 JIT 做優化。
